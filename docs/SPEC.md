# Фіксована специфікація: моніторинг доступності доменів

Документ фіксує погоджені рішення перед реалізацією. Дата: 2026-04-07.

## 1. Стек і інфраструктура

| Параметр | Рішення |
|----------|---------|
| Backend | **Laravel 12+** |
| БД | **MySQL 8+** |
| Frontend | **Vue.js 3** (інтеграція з Laravel — наприклад Vite + окремий SPA entry або Inertia; уточниться при імплементації) |
| Docker | **Так** — `docker-compose` у репозиторії |
| Git | GitHub/GitLab, ліцензія файлу **не потрібна** |

## 2. Користувачі та авторизація

- **Реєстрація:** відкрита, **без** верифікації email.
- **Логін / логіаут**, захист приватних маршрутів (middleware).
- **Ролі:**
  - звичайний користувач — бачить лише **свої** домени та перевірки;
  - **суперадмін** — доступ до всіх доменів/користувачів (обсяг UI для суперадміна: керування користувачами + перегляд/керування всіма сутностями — реалізувати в межах розумного MVP).
- **Демо:** один **спільний** тестовий акаунт (облікові дані в README / `.env.example`).

## 3. Локалізація UI

- Мови інтерфейсу: **українська** та **англійська** (перемикач / за `Accept-Language` + збереження вибору — деталі при імплементації).

## 4. Домени та перевірки

- Сутність «домен» — базовий **домен** (не складний сценарій з basic auth / кастомними заголовками / self-signed — **не** в MVP).
- Один домен може мати **кілька перевірок** (кілька URL/ендпоінтів або варіантів перевірки — модель: домен → багато **check targets** / **monitors**).
- На **кожну перевірку** окремо: метод **HTTP GET або HEAD**, інтервал, таймаут (див. нижче).

### Критерії успіху HTTP

- **Успіх:** лише відповіді з кодом **2xx**.
- **3xx:** результат у логах з **приміткою** (редирект), для статусу «доступний» **не** рахується успіхом (стан «не ОК» / окремий підтип — відобразити в UI та логах однозначно).

### SSL

- **Так:** перевірка валідності **SSL-сертифіката** (прострочений / невалідний → помилка в логі та вплив на стан перевірки).

### Налаштування за замовчуванням (де ТЗ мовчить — на розсуд розробки)

- **Таймаут HTTP:** **30 секунд** (дефолт; у моделі перевірки можна зробити поле з цим дефолтом).
- **Інтервал:** розумні межі на розсуд (наприклад 1 хв … 24 год), без жорстких вимог замовника.

## 5. Планувальник і масштаб

- Перевірки виконуються **за розкладом** (Laravel Scheduler + черга/воркер за потреби).
- Орієнтир навантаження: **сотні** доменів/перевірок — проєктувати з індексами БД та чергою, щоб не блокувати запит.
- **PaaS для демо:** обирає розробник найпростіший варіант (наприклад **Railway** або **Render**); у README — покроковий деплой і налаштування cron/worker.

## 6. Логи історії

- Зберігати: дата/час, результат (успіх/неуспіх), **HTTP-код**, час відповіді, текст помилки (якщо є), **примітки** (наприклад 3xx, SSL).
- Додатково можна зберігати короткі технічні деталі (наприклад фінальний URL після редиректу, тип помилки SSL) — **трохи розширено** порівняно з мінімальним ТЗ.
- **Не видаляти** історію автоматично — зберігати **без обмеження терміну** (MVP без retention/cleanup).

## 7. Сповіщення

- Канали: **email** та **Telegram**.
- Умова відправки: лише при **зміні стану** (наприклад up→down, down→up), не на кожну невдалу спробу.
- Конфігурація SMTP / Telegram bot token — **лише через `.env`** (без окремого UI для секретів).

## 8. Безпека та експлуатація

- **Rate limiting** на чутливі ендпоінти (логін, реєстрація, публічні форми) — **так**.
- Резервне копіювання БД на демо — **не потрібно**.

## 9. Формат здачі (нагадування)

- Репозиторій з кодом.
- **README.md** з описом, docker, деплоєм, обліковими даними демо-акаунта.
- Посилання на задеплоєну демо-версію.

## 10. Локальна розробка

- **Лише Docker** на машині розробника: PHP, Composer, Node, MySQL не встановлюються на хост (див. кореневий `README.md`).
